## 0、大模型基础认知（算法岗常问，应用岗可略）

### 0.1 预训练、SFT、RLHF 分别解决什么问题？【⭐】

| 阶段       | 解决问题                           |
| ---------- | ---------------------------------- |
| **预训练** | 模型能不能说话（学语言与知识）     |
| **SFT**    | 模型听不听话（学按指令回答）       |
| **RLHF**   | 回答好不好、安不安全（学人类偏好） |

**速记**：只有预训练 = 知识渊博但不懂人情；加 SFT = 会按指令做事；加 RLHF = 更安全。

## 1、大模型应用基础与工程选型

> **模块重要度：⭐⭐⭐**（几乎所有 LLM 应用岗都会触及）

### 1. 用一句话说明：什么时候用提示词、RAG、微调、智能体？【⭐⭐⭐】

**选型三问**：① 缺的是「知识」还是「行为」？缺知识 → RAG；缺行为 → 微调。② 要不要自动查资料、调 API、执行多步？要 → 智能体。

**速记**：提示词 = 知识已在模型内；RAG = 先检索再生成；微调 = 改行为/风格；智能体 = 大模型 + 真正执行工具。

---

### 2. RAG 和微调的主要区别与组合方式？【⭐⭐】

| 维度     | RAG                | 微调                 |
| -------- | ------------------ | -------------------- |
| 知识更新 | 改文档/索引即可    | 需重新训练或增量流程 |
| 可解释性 | 可引用片段         | 难直接溯源           |
| 典型成本 | 检索与上下文 token | 数据与算力           |
| 风险     | 检索错/上下文噪声  | 灾难性遗忘、过拟合   |

**组合**：RAG 特别适合融合新知识；微调能优化输出格式、提升复杂指令执行能力。工程上多数场景用「RAG + 提示词」或「RAG + 微调」即可；复杂系统可加 Agent 做检索路由。

**微调 vs 续训**：微调改「行为方式」，续训改「知识分布」。**顺序**：先续训再微调；多数场景 RAG + 微调即可，续训成本高。

---

### 3. 大模型应用上线前，你会关注哪些工程指标？【⭐⭐】

**参考答案**：延迟（首 token / 端到端）、可用性、错误率、token 成本、并发与限流、上下文长度、数据与隐私合规、可观测性（trace/log）、版本与回滚（模型/Prompt/索引）。

**加分**：若有 SLA/SLO，可补充 **P95/P99**、超时策略、降级（换小模型/只检索不生成）、配额与多租户隔离；安全侧补充 **审计留痕**与敏感数据脱敏。

---

### 4. 什么是「幻觉」？在应用层如何缓解？【⭐】

**定义**：模型可能「已读乱回」「已读不回」或「似是而非」——生成看似合理但与事实不符的内容。

**缓解**：RAG 通过「先查资料再回答」用真实文档约束生成；要求引用来源；结构化输出 + 校验；工具验证（计算器/DB）；不确定时拒答；人工审核；评测集回归。

---

## 2、提示词工程与模型 I/O

> **模块重要度：⭐⭐⭐**（与 `1-2`、`11`、`13`、`14` 等章呼应）

### 5. 系统提示词与用户提示词一般如何分工？【⭐⭐⭐】

- **System**：角色、边界、工具使用规范、输出格式、安全策略（稳定、少改）。
- **User / 对话**：具体任务、上下文补充、多轮澄清。

**本仓库表述**：结构化提示词 = 把稳定约束放 **System**，把动态约束放 **User**。**六要素**：角色、任务、背景、输入数据、输出格式、质量与约束；多数场景「角色 + 任务 + 输出格式」即可。**Zero-shot**：无示例直接问；**Few-shot**：先给几个示例再让模型仿写。**落地**：System 外置配置 + 版本管理。

---

### 6. 如何做 Prompt 版本管理与回归？【⭐⭐】

**要点**：与代码同仓库或配置中心管理；变更打版本；用**固定评测集**（golden questions）做回归；记录模型名、温度、top_p、模板 hash。**注意**：换底层模型后，Prompt 一般需重新调优。

---

### 7. 结构化输出有哪些实现方式？各有什么代价？【⭐⭐】

**常见路径**：JSON Schema / Function Calling 约束；解析器二次校验（如 Pydantic）；失败重试；必要时「先输出 JSON 再解析」的专用小模型。**本仓库**：提示词六要素中的「输出格式」可指定表格、JSON、代码等；LCEL 链中 `prompt | model | parser` 的 parser 负责解析。

**权衡**：约束越强，**失败重试**概率可能上升；需在「格式可靠」与「体验/延迟」间平衡。

## 3、上下文窗口、长上下文与记忆架构

> **模块重要度：⭐⭐⭐**（公开面试资料中极高频：**Context vs Memory**、长对话、成本）

### 8. 「上下文窗口」和「记忆」有什么区别？工程上如何分工？【⭐⭐⭐】

**上下文窗口**：模型单次推理能看到的 token 上限，按 token 计费、偏贵。**记忆**：应用层存储对话历史的组件，让 AI 能「记住」之前的对话。

**本仓库实现**：读历史 → 拼入 Prompt → 调模型 → 写回历史。用 RunnableWithMessageHistory + BaseChatMessageHistory（内存版 InMemory、持久化版 Redis）。**结论**：大块、长期、要检索的放外部记忆；窗口里只放高信号、短而准的上下文。

---

### 9. 什么是「Lost in the Middle」？对 RAG 有什么启示？【⭐⭐】

**现象**：长上下文中，模型对**开头和结尾**利用更好，**中间**关键证据容易被淹没（U 型注意力）。

**启示**：控制进窗片段数量、优先 rerank 再组装；重要结论放靠前或靠后；压缩/摘要减少中间噪声；必要时二次检索。

---

### 10. 超长多轮对话如何兼顾「不断档」与「省 token」？【⭐⭐】

**常见组合拳**：

1. **滑动窗口**：只保留最近 N 轮；风险是丢掉早期约束，需配合摘要。
2. **周期性摘要（Checkpoint Summary）**：每 K 轮把历史压成结构化摘要（目标、已确认事实、待办、用户偏好）。
3. **向量检索历史**：把旧轮次写入向量库，按当前问题动态召回相关片段，而非全量塞进窗口。
4. **结构化记忆槽**：如用户城市、会员等级等写入键值表，减少靠模型「回忆」。

---

### 11. 有了「超长上下文模型」，还需要 RAG 吗？【⭐】

**通常仍需要**，理由可答：

- **成本与时延**：全量塞文档很贵、很慢；RAG 可先缩小证据集。
- **时效与权限**：企业知识分区、按租户过滤，RAG + metadata 更自然。
- **可溯源与合规**：引用片段、审计「用了哪段材料」比「全在窗口里」更易治理。

**补充**：长窗口更适合「**少量长文档内推理**」；企业海量知识仍多靠检索缩小范围。

## 4、RAG 与知识库

> **模块重要度：⭐⭐⭐**（对应 `2-RAG…`、`18`、`19`）

### 12. 描述一个完整的 RAG 流水线及关键模块。【⭐⭐⭐】

**标准流程**：

1. **索引阶段**：文档加载器 → 文本分割器（分块）→ 嵌入模型向量化 → 向量数据库写入。
2. **检索阶段**：用户提问 → 问题向量化 → 相似性检索（top_k）→ 取回相关片段。
3. **生成**：检索片段 + 用户问题 → 填入提示词模板 → 大模型生成 → 后处理（引用、脱敏）。

**核心**：RAG = 先查资料再作答，用真实文档约束生成、减轻幻觉。**追问**：chunk 策略、metadata 过滤、多租户隔离、索引更新与一致性。

---

### 13. RAG 效果不好时，你会如何排查？【⭐⭐】

**分层排查**：

1. **检索层**：查不到 / 召回噪声大 → 调 chunk、embedding、索引参数、hybrid、rerank、query 改写。
2. **上下文组装**：token 截断导致丢证据 → 压缩、摘要、分层检索；注意 **「Lost in the Middle」**（见 **9**）——无关片段过多时，关键证据即使在上下文中也可能被模型忽略。
3. **生成层**：不看上下文瞎编 → 强约束引用、降低温度、加「无证据则拒答」。
4. **数据层**：文档脏、过期、权限错误。

**RAG 常见 10 大坑（面试可脱口而出）**：

| 问题         | 典型原因               | 解决思路                               |
| ------------ | ---------------------- | -------------------------------------- |
| 内容缺失     | 切片不合理、召回率低   | 调分段长度、多向量检索、增量索引       |
| 错过重要文档 | 距离计算/排序不佳      | 优化相似度、引入 rerank                |
| 脱离上下文   | 多文档拼接语义丢失     | 语义加权拼接、Context Re-weighting     |
| 未能提取答案 | 阈值不当、Prompt 弱    | 调相似度阈值、强化「必须基于引用回答」 |
| 格式错误     | 数据清洗差、输出未约束 | 统一索引格式、Prompt 约束输出模板      |
| 领域特异性差 | 通用 embedding 不适配  | 领域词向量、知识图谱增强               |
| 回答不全面   | 召回范围小             | 多通道检索（keyword + embedding）      |
| 数据处理慢   | 非结构化提取慢         | 批量向量化、流式索引                   |
| 结构化数据弱 | 表格/SQL 支持差        | LLM 生成 SQL 再执行                    |
| PDF 提取难   | 表格、版式复杂         | 布局识别 + OCR + 坐标级切分            |

---

### 14. RAG 常见评测维度有哪些？【⭐⭐】

**检索**：Recall@K、MRR、nDCG（视工具链而定）。  
**生成**：Faithfulness（是否忠于检索内容）、Answer Relevance、Citation 正确率。  
**系统**：端到端延迟、成本、用户满意度。

**工具链提示**：开源/商用里常提到 **RAGAS、DeepEval、TruLens** 等框架做自动化指标；面试答出「检索指标 + 忠实度/相关性 + 端到端回归集」即可，不必死记单个库 API。

---

### 15. 稀疏检索与稠密向量检索如何取舍？Hybrid 适用场景？【⭐】

**简述**：稀疏（BM25）擅长大致关键词匹配；稠密擅长度义相似与 paraphrase。  
**Hybrid**：企业文档、混合中英文、专有名词多、用户 query 口语化时，常能稳态提升召回。

## 5、RAG 进阶：分块、GraphRAG、Agentic RAG

> **模块重要度：⭐ ～ ⭐⭐**（进阶岗/架构岗常问；与 `19` 章 RAG 实战可对照）

### 16. 文档分块（Chunking）有哪些策略？各有什么坑？【⭐⭐】

常用 **RecursiveCharacterTextSplitter**（按字符递归切分，尽量保持语义完整）。

| 策略           | 优点           | 典型坑                         |
| -------------- | -------------- | ------------------------------ |
| 固定字符/Token | 实现简单       | 切断表格/代码/条款，语义不完整 |
| 按标题/结构    | 边界较自然     | 依赖源文档结构质量             |
| 语义分块       | 语义完整性好   | 实现复杂、耗时                 |
| 滑动窗口+重叠  | 减少边界信息丢 | 存储与召回冗余上升             |

**面试收尾**：分块不是一次定终身，应用 **评测集 + badcase** 迭代；表格/代码常要 **专门解析器** 而非纯文本切分。

---

### 17. GraphRAG（图谱增强）适合什么场景？与传统向量 RAG 怎么选？【⭐】

**适合**：强 **实体-关系**、需要 **多跳推理**、社区/子图总结（如「某组织相关人物网络」）类问题。  
**代价**：建图、更新、存储与工程复杂度更高。

**传统向量 RAG**：适合大量非结构化段落、以「找相似段落」为主的问答。  
**实践**：可先向量 RAG 上线，确有大量多跳关系痛点再评估图谱路线。

---

### 18. 什么是 Agentic RAG？和「检索一次 + 生成」有何不同？【⭐⭐】

**Agentic RAG**：由 Agent 在循环中自主决定**是否检索、检索什么、是否换 query 再检**，甚至对检索结果自检后再检。**传统 RAG**：固定「检索一次 → 生成」单向管道。

**区别**：从单向管道变为带反馈的闭环（常与 ReAct/LangGraph 结合）。**风险**：更多模型轮次 → 成本、延迟、失控概率上升，必须配步数/预算/护栏。

---

### 19. RAG 有哪些常见「反模式」？【⭐】

- chunk 过大（噪声淹没）或过小（语义不全、召回碎片化）。
- 只向量检索不做 **hybrid/rerank**，专有名词场景召回差。
- **embedding 与业务语种/领域**不匹配却不评测。
- **K 过大** 填满上下文，触发 **lost in the middle**。
- 无 **metadata/ACL**，跨租户串料（严重事故）。

---

## 6、向量检索与 Embedding 选型

> **模块重要度：⭐**（工程面试常有一题；与 `18` 章呼应）

### 20. 向量数据库选型你会看哪些维度？【⭐】

**本仓库** 多用 Redis（Redis Stack）做向量存储。

**选型维度**：数据规模与 QPS、是否需要 metadata 过滤、延迟与可用性、多副本/云托管、成本、生态。LangChain 抽象为 VectorStore，具体有 Chroma、Pinecone、FAISS、Redis 等实现。

---

### 21. Embedding 模型与业务不匹配会怎样？如何验证？【⭐】

**现象**：召回差、跨语种漂移、领域词（医疗/法律/工单）向量距离不合理。

**应对**：在 **真实 query‑doc** 上抽评测集看 Recall/MRR；必要时换域内 embedding、微调向量模型、或对查询做 **同义词扩展/hybrid**。

## 7、框架选型与其他 SDK

> **模块重要度：⭐⭐**

### 22. LangChain、LangGraph、自研、低代码平台如何选型？【⭐⭐】

**LangChain vs Coze/Dify**（本仓库 9 章）：Coze/Dify 是**低代码平台**，拖拽搭建；LangChain 是**代码框架**，深度集成、强定制。二者可并存：平台做原型、对外应用；LangChain 做内部工具、复杂逻辑。

- **快速原型 / 标准工具链**：LangChain。
- **强状态、循环、人机协同、复杂分支**：LangGraph。
- **业务方主导、少研发**：Coze/Dify。
- **强管控与深度定制**：自研编排 + 统一治理层。

## 8、LangChain 与 Agent 运行时

> **模块重要度：⭐⭐⭐**（对应 `9`～`17`、`21`）

### 24. 如何使用 LangChain 开发一个 Agent？（原理级）【⭐⭐⭐】

**本仓库 V1.0 写法三步**：

1. **定义工具**：`tools = [get_weather]`，用 `@tool` 或封装函数，写清 name、description、参数 schema。
2. **构建 Prompt**：`system_prompt = "You are a helpful assistant."`。
3. **调用 create_agent**：`create_agent(model, tools, system_prompt)` 得到 agent，再 `agent.invoke({"input": "..."})`。

**原理**：Agent 通过 **ReAct 循环**（Thought → Action → Observation）完成「推理 → 行动 → 观察」直到结束。**Agent** 负责决定调哪个工具、何时结束；**AgentExecutor**（或 V1.0 底层 LangGraph）负责真正执行工具并把结果传回。**案例**：`AgentSmartSelectV1.0.py`、`AgentReact.py`。

---

### 25. LangChain Agent V0.3 与 V1.0 的主要差异是什么？【⭐⭐】

| 维度     | V0.x（如 AgentSmartSelectV0.3.py）                      | V1.0（如 AgentSmartSelectV1.0.py）                        |
| -------- | ------------------------------------------------------- | --------------------------------------------------------- |
| 创建方式 | Prompt 模板 + create_tool_calling_agent + AgentExecutor | 直接 `create_agent(model, tools, system_prompt)` 一步到位 |
| 提示词   | 必须用 PromptTemplate，含 agent_scratchpad              | 传字符串即可                                              |
| 调用     | `agent_executor.invoke`                                 | `agent.invoke`                                            |
| 底层     | 传统链                                                  | LangGraph 状态机                                          |

**面试可说**：本仓库有 V0.3 与 V1.0 同一业务（多城市天气比较）的对比案例，可现场演示差异。

---

### 26. 为什么「只有 Tool 不够，还需要 Agent」？【⭐⭐】

**核心**：**Tool = 能力**，**Agent = 决策 + 使用这些能力**。Tool 只提供「能做什么」，不负责「何时做、按什么顺序做、做到什么程度为止」。

**本仓库举例**：单步查天气 → 直接 Tool；「订明天北京到上海机票 + 查上海天气」→ Tool 不会自己决定先订票还是先查天气，需要 Agent 推理并编排；「PDF 总结 + 发邮箱」→ 需多个 Tool，谁先谁后要 Agent 决定。

## 9、LangGraph 与复杂工作流

> **模块重要度：⭐⭐⭐**（对应 `22`～`25`）

### 27. 在 LangGraph 中如何处理 Agent 的工具调用？【⭐⭐⭐】

- **节点化**：LLM 节点、工具节点分离，职责清晰。
- **条件边**：`add_conditional_edges` 根据模型输出（是否 tool call、哪条路由）跳转。
- **状态**：`StateGraph` + `TypedDict`/schema 管理共享状态，便于持久化与回放。
- **循环控制**：显式定义继续调用工具或结束的边条件，避免隐式死循环。

**为何用 LangGraph**（本仓库 22 章）：Chain 太线性，难以处理循环与分支；Agent 太黑箱，难以控制。LangGraph 把工作流抽象为有向图，在可控性与表达能力之间取得平衡。**案例**：`案例与源码-3-LangGraph框架/`。

---

### 28. LangGraph 相比传统 workflow 引擎的优势？【⭐⭐】

**本仓库**：`01-helloworld`、`02-basic_graph` 等入门案例。

| 维度        | LangGraph                          | 传统固定工作流     |
| ----------- | ---------------------------------- | ------------------ |
| 动态性      | 模型在环中实时决策                 | 流程多预定义       |
| 状态        | 结构化、可持久化                   | 常依赖隐式上下文   |
| 循环 / 分支 | 原生表达                           | 往往需要硬编码绕路 |
| 人机协同    | `interrupt`、checkpoint 等一等公民 | 多需自建           |

**本质**：LangGraph 把 **LLM 作为控制流的一部分**，同时保留工程可控性。

---

### 29. LangChain / LangGraph 企业落地的常见挑战？【⭐⭐】

1. **可观测性**：多跳调用要 trace（每步输入输出、工具参数）。
2. **版本治理**：Prompt、工具 schema、图结构需版本化与灰度。
3. **延迟与成本**：多轮模型调用；需缓存、路由小模型、并行化。
4. **容错**：工具失败、格式错误、重试与降级策略。
5. **测试**：除单元测试外，需集成评测与回放固定会话。

---

### 30. Checkpoint、HITL 在工程上的价值？【⭐】

**Checkpoint**：持久化状态，长任务中断后可恢复、便于 replay 排错。**HITL**：敏感节点（支付、删数据）插入人工审批，确认后再继续。

## 10、低代码智能体平台（Coze / Dify）与工作流

> **模块重要度：⭐⭐**（对应 `3`、`4`～`7` 及案例；**业务落地岗**很爱问）  
> _注：本仓库以平台实战为主，下列题为通用面试抽象。_

### 31. Coze / Dify 这类平台与自研 LangChain/LangGraph 的边界怎么划分？【⭐⭐】

**平台优势**：可视化编排、权限与多租户、内置知识库/插件生态、上线快。  
**自研优势**：强定制流程、深度适配内部系统、复杂状态机/审计、与现有 CI/CD 一体化。

**选型**：POC / 业务迭代快 → 平台；核心链路长、合规与可编程要求高 → 代码化框架 + 自建治理。**本仓库**：3.1 ～ 3.10 为 Coze/Dify 案例（行业调研 PPT、客服分析、商品评论等）；4、5 章讲 API 调用与部署。

---

### 32. 用工作流做「智能体」时，和真正的 LLM Agent 有什么差别？【⭐⭐】

**工作流**：多数节点是**确定性**的，分支由规则或少量模型节点驱动，易测试；执行流程固定、结果可控。
**Agent**：模型在环中**反复决策**是否调工具，路径更动态，需更强的护栏与观测。

**实践**：复杂业务常采用 **「工作流编排 + 局部 Agent」** 的混合架构。Dify、Coze 等平台将工作流与 Agent 作为两种应用形态。

---

### 33. 如何通过 API 将平台工作流接入自有系统？【⭐】

**要点**：鉴权（API Key / OAuth）、异步与 webhook、幂等与重试、入参 schema 版本化、限流与超时；生产要记录 **run_id** 便于对账与排错。**本仓库**：第 4、5 章有 Dify/Coze 的 Python 调用示例；第 6、7 章为部署说明。

## 11、Function Calling / Tool Use

> **模块重要度：⭐⭐⭐**（对应 `17-Tools工具调用`）

### 34. Function Calling 的基本原理？【⭐⭐⭐】

**LLM 只负责输出「要调哪个工具、传什么参数」**，真正执行的是你的代码。

**五步流程**（本仓库 1-3 章有 DeepSeek API 演示）：① 告诉模型有哪些工具、用户说了什么；② 模型返回 tool_calls（name + arguments）；③ 代码执行工具拿到结果；④ 把结果塞回对话发给模型；⑤ 模型根据结果生成最终回复。

**本质**：模型产**调用计划**，**宿主负责真实世界副作用**与安全。工具定义用 `@tool`，参数可用 Pydantic；绑定模型用 `bind_tools`。

---

### 35. Function Calling 与「让模型输出 JSON 再自己解析」有何区别？【⭐⭐】

**原生 FC**：协议级约束、可靠性高、一次推理即可、平台/SDK 支持好。**纯文本 JSON**：易受话术干扰、需解析与修复、可能重试。**结论**：生产优先原生能力。

---

### 36. 在 RAG 系统里，Function Calling 可以做什么？【⭐⭐】

- Query 改写与多查询扩展；路由到不同知识库/工具；检索后处理（过滤、排序）；判断是否需要二次检索；调用计算器/结构化查询等。

**价值**：把固定 pipeline 变为按上下文动态组合的能力；由模型/Agent 决定检索策略而非固定「检索一次 → 生成」（见 Agentic RAG）。

---

## 12、工具生态：幻觉、规模化与推理参数

> **模块重要度：⭐⭐**（Agent 落地与排错高频）

### 38. 什么是「工具幻觉」？如何缓解？【⭐⭐】

**含义**：模型调用不存在的工具、捏造参数、或在不允许写操作时强行走写路径（发生在 tool 空间）。

**缓解**：宿主白名单校验工具名；参数 JSON Schema/Pydantic 校验；工具描述清晰；过多工具时分层/分域或工具检索；高风险工具强制确认；失败时明确错误回传让模型重试（限次）。**生产**：意图识别（小模型/规则）→ 工具匹配（意图 →API 注册表）→ 置信度低时追问；异常回传模型 Self-Correction，连续失败转人工。

---

### 39. 工具数量很多时，如何控制上下文与稳定性？【⭐⭐】

**思路**：

1. **分层**：Supervisor 先路由到子域，只在子域内暴露少量工具。
2. **动态工具选择（Tool Retrieval）**：先从库中检索 Top‑M 个工具描述进 Prompt，而非全量列出。
3. **聚合工具**：把细粒度 API 包装成少量「业务动作」工具，减少选择困难。
4. **文档与命名**：工具名可读、描述含「何时用/何时不用」，降低误选。

---

### 40. Temperature、top_p 在 Agent 系统里一般怎么配？【⭐】

**经验法则（非绝对）**：

- **选工具、抽参数、遵循严格格式**：偏低 temperature，减少随机胡选。
- **开放创作、营销话术**：可适当升高。
- **生产环境**：常按 **场景分配置**（路由/工具调用/最终润色分开），并 **固定 seed**（若平台支持）便于复现问题。

## 13、Agent 基础概念与设计

> **模块重要度：⭐⭐⭐**

### 41. Agent 和传统「单轮问答式」AI 应用有何不同？【⭐⭐⭐】

| 维度     | Agent 应用         | 传统问答         |
| -------- | ------------------ | ---------------- |
| 决策     | 可自主选用工具     | 多为一问一答     |
| 交互     | 多步、目标导向     | 常无状态或弱状态 |
| 能力扩展 | 插件化工具         | 功能固定         |
| 风险     | 工具滥用、成本失控 | 相对较低         |

**本仓库表述**：大模型只会「说」怎么调接口；智能体 = 大模型 + 真正去**执行**工具并把结果塞回对话。**Tool** 像工具箱里的螺丝刀、锤子；**Agent** 像有判断力的工匠——知道什么时候用、按什么顺序用。

---

### 42. Agent 适合解决哪类复杂任务？【⭐⭐】

**任务特征**：多步骤、信息在外部、需要试错与调整、存在分支与回溯。

**能力匹配**：

- **分解子任务**：将复杂目标拆成可执行步骤，而非一次性生成。
- **工具使用**：按需调用搜索、计算、API、数据库等外部能力。
- **迭代修正**：根据工具返回或用户反馈调整策略，而非一次定生死。
- **记忆与上下文延续**：跨轮次维护任务状态、用户偏好、中间结果。

**典型场景**：订票+查天气+汇总、多文档分析+邮件发送、研究类多源检索+报告生成。单步、规则清晰的任务用 Tool 或工作流即可，不必上 Agent。

---

### 43. 一个生产级 Agent 架构应包含哪些模块？【⭐⭐】

**核心模块**：感知（接收输入、意图粗分）→ 规划（ReAct/ToT 任务分解）→ 记忆（Redis 短期 + DB 长期）→ 工具（FC/MCP）→ 执行与反馈（工具执行器、错误处理）→ 评测与监控（trace、指标）→ 安全策略（白名单、HITL、脱敏）。

**落地要点**：记忆分层；规划有步数上限；工具调用审计留痕。

---

### 44. 如何设计 Agent 的可扩展性？【⭐⭐】

**工具层**：工具即插件，新能力通过注册新 Tool 实现，无需改核心逻辑；支持 MCP 动态发现。

**配置层**：Prompt、温度、步数上限、工具列表等外置配置或配置中心，支持灰度与 A/B。

**架构层**：无状态设计，会话状态存 Redis/DB，支持多实例水平扩展；复杂域拆为 **多 Agent**（Supervisor + 子 Agent），各司其职。

**治理层**：统一观测与注册中心，工具/Agent 版本化，便于回滚与问题复现。

---

### 45. 如何在 Agent 中增强推理能力？【⭐⭐】

- CoT；**ReAct**（Thought-Action-Observation）；Reflection；ToT；或 **LangGraph** 显式状态机降低「乱跑」概率。

**提醒**：推理链越长，**成本与延迟**越高，需上限控制。

## 14、多智能体与 A2A

> **模块重要度：⭐ ～ ⭐⭐**（对应 `26-LangGraph多智能体与A2A`；**架构岗/大厂**问得多）

### 46. MCP 和 A2A 有什么本质区别？【⭐⭐】

- **MCP**：关注**工具访问**，定义大模型如何与工具、数据、资源交互；像「工具车间」——让工人知道每个工具的位置和用法。
- **A2A**：关注**代理协作**，定义智能体之间如何发现、通信、协作；像「会议室」——让不同专业代理能坐在一起协调任务。

**修车厂类比**：MCP 让维修工知道如何使用千斤顶、扳手；A2A 让客户能与维修工沟通，并让维修工之间或与配件商代理协作。

---

### 47. 多智能体有哪些常见拓扑？各自适用场景？【⭐⭐】

| 拓扑                    | 特点                        | 适用场景                                    |
| ----------------------- | --------------------------- | ------------------------------------------- |
| **单 Agent**            | 一个 LLM + 一组工具         | 简单对话、单一领域（天气、翻译、知识库 QA） |
| **Network**             | 多 Agent 平等、彼此可通信   | 多视角协作、并行检索汇总、研究讨论          |
| **Supervisor**          | 主控 Agent 调度子 Agent     | 企业助手（IT/HR/财务）、客服分诊            |
| **Supervisor as tools** | 主模型把子 Agent 当工具调用 | 子 Agent 像专业插件                         |
| **Hierarchical**        | 多层监督者                  | 大型任务拆解、复杂管道                      |
| **Custom**              | 自由组合 + HITL             | 强合规、需人工审批                          |

**本仓库案例**：`SupervisorV0.3.py`、`SupervisorV1.0.py`、`SupervisorHandoff.py`；A2A 案例 `Agent2Agent.py`（携程订机票、美团订酒店、滴滴打车，总协调按序调用）。

## 15、MCP（模型上下文协议）体系

> **模块重要度：⭐⭐⭐**（对应 `20-MCP模型上下文协议`；2025 年面试显著增多）

### 49. MCP 解决什么问题？与 Function Calling 的关系？【⭐⭐⭐】

**痛点**：各框架各自实现 Function Calling，缺乏统一标准；工具复用难。**MCP**：统一协议，规定工具如何暴露、如何被多端复用——可类比「AI 世界的 USB-C」。

**关系**：Function Calling 是「模型会调工具」的底层能力；MCP 是协议层，规定工具如何暴露与调用。二者组合：MCP 管理工具，模型通过 FC 触发。**本仓库**：`McpServer.py`、`McpClientAgent.py`。

---

### 50. MCP 的典型架构角色？【⭐⭐】

| 角色              | 说明（本仓库 20 章）                            |
| ----------------- | ----------------------------------------------- |
| **MCP 主机**      | 发起请求的 AI 应用（聊天机器人、AI 驱动的 IDE） |
| **MCP 客户端**    | 在主机内，与 MCP 服务器 1:1 连接                |
| **MCP 服务器**    | 为客户端提供上下文、工具和提示                  |
| **本地/远程资源** | 服务器可访问的文件、数据库或 API                |

**传输模式**：**STDIO**（本地、标准输入输出）适合单机、轻量；**SSE**（HTTP 长连接）适合独立服务、多客户端。**本仓库**：`McpServer.py`、`McpServerByFastMCP.py` 演示工具注册与暴露；`mcp.json` 配置多服务连接。

---

### 51. MCP 常见传输方式及场景？【⭐⭐】

| 维度         | SSE（Server-Sent Events）             | STDIO                                  |
| ------------ | ------------------------------------- | -------------------------------------- |
| **典型场景** | 独立部署的 MCP 服务、长连接或流式推送 | 本地集成、命令行工具、与主进程同机部署 |
| **传输方式** | HTTP 长连接（Keep-Alive）             | 操作系统标准输入/输出（stdin、stdout） |
| **数据方向** | 服务端 → 客户端（单向推送）           | 双向流                                 |

**一句话**：SSE 适合「独立服务、多客户端、网络访问」；STDIO 适合「单机、轻量、进程内或本地管道」。**本仓库**：`McpServerByFastMCP.py` 用 STDIO；天气 SSE 版用 `transport="sse"`。

---

### 52. 开发 MCP Server 的一般流程？【⭐⭐】

1. **创建实例**：`mcp = FastMCP("服务名")`。
2. **注册工具**：`@mcp.tool()` 装饰函数，写清 name、description、参数类型。
3. **注册资源/提示**（可选）：`@mcp.resource(uri)`、`@mcp.prompt()`。
4. **启动**：`mcp.run(transport="stdio")` 或 `transport="sse"`。

**本仓库**：`McpServer.py`、`McpServerByFastMCP.py`、`McpServerWeatherByFastMCP.py` 可直接对照。

---

### 53. Agent 如何接入 MCP 工具（思路级）？【⭐⭐】

**本仓库流程**：`load_servers(mcp.json)` → `MultiServerMCPClient(connections)` → `async with client.session(): tools = client.get_tools()` → 交给 `create_tool_calling_agent` + `AgentExecutor`，与 V0.3 用法一致。**案例**：`McpClientAgent.py`。工具定义与运行在 MCP 服务端，客户端按协议发现并调用。

## 16、Agentic 系统与交互体验（综合表述）

> **模块重要度：⭐⭐**（系统设计题「开场白」很好用）

### 58. 什么是 Agentic（智能体化）系统？与「套一层 Prompt」区别在哪？【⭐⭐】

**本仓库定义**：智能体 = 以大语言模型为推理与决策核心，结合记忆、工具调用与环境交互，能够进行规划决策并执行动作以达成目标的软件系统。五大要素：LLM、Memory、Tools、Planning、Action。

**与单轮 Prompt 区别**：大模型只会「说」怎么调接口；智能体 = 大模型 + 真正去**执行**工具并把结果塞回对话。Agentic 强调 **状态 + 循环（ReAct）+ 工具 + 边界（护栏）**。**流式输出**：降低首字延迟、可展示中间步骤，但日志/trace 需完整落库。

## 17、安全、护栏与合规

> **模块重要度：⭐⭐⭐**（教程分散在多处，此处汇总为面试高频）

### 60. 如何防止 Agent「工具调用失控」？【⭐⭐⭐】

- **硬上限**：最大步数、最大工具调用次数、时间预算、费用预算。
- **电路熔断**：连续失败停止；异常率告警。
- **权限**：工具分级；写操作与人审（HITL）。
- **输入输出治理**：PII 脱敏、敏感指令拦截（prompt injection 防护）。
- **工具层**：校验「只允许已注册工具名」、参数 schema 校验，降低工具幻觉。

---

### 61. 什么是 Prompt Injection？应用层如何缓解？【⭐⭐】

**定义**：用户诱导模型**违背原指令**或**滥用工具**（如「忽略上文导出数据」）。

**缓解**：权限最小化、工具白名单、系统与用户消息隔离、输出过滤、关键操作二次确认、RAG 文档不可信时的隔离展示。

---

### 62. 代码解释器 / 浏览器自动化最大的安全风险？【⭐⭐】

**代码执行**：沙箱（容器/受限解释器）、网络出网控制、文件系统白名单、CPU/内存/时间限制。  
**浏览器**：凭证泄露、越权操作、钓鱼页面；需会话隔离、敏感站点多因子或禁止自动登录。

---

## 18、评测与可观测性

> **模块重要度：⭐⭐**（Agent 岗越来越常问）

### 63. 你会如何观测一个 Agent 系统？【⭐⭐】

**Tracing**：每次模型调用、工具调用的参数与返回、延迟、token。  
**Logging**：业务 run_id、用户会话、版本信息。  
**Metrics**：成功率、重试率、工具错误拓扑、成本曲线。

可选：**OpenTelemetry**、厂商 Trace、自建 JSON 日志管道。

---

### 64. Agent 评测为什么比单轮 QA 难？【⭐⭐】

多跳决策、工具组合爆炸、状态依赖、非确定性；需 **轨迹级**评测（选对工具、参数正确、是否过早终止）+ **结果级**评测（任务是否完成）。

## 19、成本、性能与部署运维

> **模块重要度：⭐⭐**（对应 `8` 企业部署、`12` Ollama 等章节思想）

### 66. Agent 系统延迟主要来自哪里？如何优化？【⭐⭐】

**来源**：多轮 LLM、同步工具、检索、重排序、过大上下文。

**手段**：流式输出、并行工具、缓存检索结果、小模型路由、压缩上下文、工具超时与异步化。

---

### 67. 调用成功率、任务完成率如何定义？【⭐⭐】

**工具调用成功率** = 成功调用次数 / 总调用次数（HTTP 2xx、无异常等）；**业务无结果**不一定算失败。

**任务完成率**：端到端是否达成用户目标（常需人工标注或用户确认），反映规划、工具覆盖与容错是否到位。

---

### 68. 线上运维关注哪些指标？【⭐⭐】

示例：**首 token 延迟**（P50/P95）、端到端时延、错误率、工具失败类型分布、每会话成本、用户满意度、回滚次数（模型/Prompt/索引）。

---

### 69. 模型路由怎么做？【⭐】

**分层策略**：

- **意图分类**：用小模型（如 7B）或规则（关键词/正则）做粗分，决定走 FAQ / RAG / Agent / 人工。
- **复杂度路由**：简单问答、格式固定 → 小模型；多步推理、开放生成 → 大模型。
- **成本优化**：热点问题缓存；能检索直接回答的不调用生成；按用户等级/场景分配配额。

**落地**：建立「意图 → 模型/流程」映射表，配合 A/B 测试调优；生产需监控各路由分支的延迟与成本占比。

## 20、垂直场景：Deep Research

> **模块重要度：⭐**（研究/报告类产品岗）

### 70. 如何设计 Deep Research 类系统？与单轮 RAG 有何不同？【⭐⭐】

**主流程**：查询规划（分解子问题）→ 多源检索（Web/DB/知识库）→ 证据融合（去重、冲突检测）→ 迭代补充检索 → 结构化报告（摘要/细节/引用）。**控制器**：根据中间结果决定继续检索还是进入写作。

**与传统 RAG 区别**：检索多轮自适应 vs 单次；多源异构 vs 单一知识库；长文+引用 vs 短答。**可信度**：多源交叉验证、强制引用、关键结论 HITL、对「找不到证据」显式说明。

## 21、垂直场景：NL2SQL Agent

> **模块重要度：⭐ ～ ⭐⭐**（数据智能、BI 方向）

### 74. 如何构建 NL2SQL Agent？如何保证正确性与安全？【⭐⭐】

**流水线**：Schema Linking（找表字段）→ SQL 生成 → 语法校验 → 沙箱/只读执行 → 结果解释 → 置信度与澄清。**Agent 优势**：多轮澄清、执行失败自愈、结合业务词典。

**安全**：Schema 约束、SQL 解析器校验、黑名单/白名单、只读账号、LIMIT、行级权限、Self-correction 限次重试。**歧义处理**：列名多义 → 列出候选让用户选择；时间模糊 → 反问具体区间；置信度低 → 展示预览请用户确认后再执行。**原则**：宁可多问一句。

## 22、垂直场景：智能客服

> **模块重要度：⭐⭐**（对应多个 Coze/Dify 客服案例）

### 78. 智能客服系统典型架构？高并发如何优化？【⭐⭐】

**链路**：意图识别 → 路由（FAQ / RAG / 工单 / 人工）→ 多轮澄清 → 回复与质检。**模块**：知识引擎、会话状态、人机协同、反馈闭环。**相对 FAQ 提升**：语义理解、多轮对话、个性化、复杂问题解决率更高。**RAG 价值**：FAQ 未命中时从手册/工单检索兜底；政策变更时更新索引即可。

**高并发**：水平扩展无状态服务、热点缓存、向量检索缓存、小模型做意图、大模型仅用于复杂生成、流式降低感知延迟。

## 23、垂直场景：代码解释器与浏览器

> **模块重要度：⭐**（数据分析/运维岗）

### 82. 如何设计安全的代码解释器？【⭐⭐】

沙箱（容器/受限运行时）、资源限额、网络策略、模块白名单、超时、输出大小限制、审计。**与本地 REPL 区别**：隔离性、可观测、与 LLM 工具链集成。

### 83. 浏览器自动化 Agent 的分层？如何提高可靠性？【⭐⭐】

**分层**：控制层（Playwright）→ 认知层（理解页面与任务）→ 规划层（动作序列）→ 验证层（截图/DOM 断言）。**可靠性**：重试、超时、截图存证、关键步骤断言、连续失败转人工或降级。

## 附录 A：速记表

### A1 ReAct 循环

**Thought → Action → Observation** 直到满足结束条件。四步：思考（形成计划）→ 行动（选工具、传参）→ 观察（接收工具返回）→ 循环或结束。

### A2 失败模式清单（面试常用来「展示经验」）

目标含糊、工具描述不清、检索噪声、工具返回非结构化、无步数上限、无权限隔离、无观测导致不可复盘。

### A2.5 本仓库核心概念速记

| 概念                 | 主要作用                                              |
| -------------------- | ----------------------------------------------------- |
| **Tool**             | 能力的封装，大模型能「用」的原子能力                  |
| **Function Calling** | 大模型「决定调哪个、传什么参数」并解析返回的底层能力  |
| **RAG**              | 通过检索把相关文档注入 prompt，让模型「看到」更多信息 |
| **MCP**              | 规定工具/数据源如何暴露、如何被发现的标准化协议       |
| **Agent**            | 负责何时用哪个 Tool、按什么顺序、何时结束的决策层     |

### A2.1 上下文相关速记

**Context = 当前窗口里的高代价工作记忆；Memory = 外部持久化 + 可检索。** 长上下文 ≠ 不用 RAG；注意 **lost in the middle** → 控制进窗质量优于盲目加 K。

### A3 推荐答题结构（系统设计题）

**场景与约束** → **数据流与状态** → **关键算法/模型** → **工具与权限** → **评测与监控** → **成本与降级**。

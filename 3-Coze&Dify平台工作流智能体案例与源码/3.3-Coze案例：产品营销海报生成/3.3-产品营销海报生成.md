## 4. 分析工作流

此处可以看到两个工作流

![](../../images/3.3/3.3-4-1-1.png)

### 4.1 img_generate

这是关键的生成图片的逻辑

#### 1. 开始

![](../../images/3.3/3.3-4-1-2.png)

#### 2. 导出链路判定

![](../../images/3.3/3.3-4-1-3.png)

根据 logo_place 判断是否为导出链路，前者为空则不是导出，反之亦然。

##### 2.1 生图链路

logo_place 为空说明是生图链路

**判断是否为文生图链路**

![](../../images/3.3/3.3-4-1-4.png)

如果 img 参数为空则说明是文生图，否则是图生图

###### 2.1.1 文生图

根据具体的宽高比生成图片

**首先由一个分支节点根据不同的 propotion 路由至不同的生图节点**

![](../../images/3.3/3.3-4-1-5.png)

**然后在不同的分支选择不同的生图比例，并选好生图模型，将用户的提示词作为生图提示词传入。**

以 2:3 为例

![](../../images/3.3/3.3-4-1-6.png)

此处的生图节点都是图像处理下的图像生成节点

![](../../images/3.3/3.3-4-1-7.png)

###### 2.1.2 改图

如果 img 不为空则是改图。

根据提示词判定是改图还是变清晰，如果提示词不为空是改图，否则为变清晰

![](../../images/3.3/3.3-4-1-8.png)

**改图**

将原图和提示词填入

![](../../images/3.3/3.3-4-1-9.png)

改图节点选择的是指令编辑插件

![](../../images/3.3/3.3-4-1-10.png)

**变清晰**

将原图填入

![](../../images/3.3/3.3-4-1-11.png)

变清晰节点选择的是画质提升插件

![](../../images/3.3/3.3-4-1-12.png)

##### 2.2 导出链路

###### 2.2.1 添加二维码

**根据 qr_img 参数是否为空判断用户是否上传了二维码**

![](../../images/3.3/3.3-4-1-13.png)

① 如果用户上传了二维码

**调整用户二维码图片大小，最大尺寸不超过 200 像素**

![](../../images/3.3/3.3-4-1-14.png)

**添加用户二维码**

![](../../images/3.3/3.3-4-1-15.png)

将调整后的用户二维码作为上层图，将传入的图片作为底图，上层图已经经过了尺寸调整，缩放比例为 100%，即不缩放。并根据 qr_place 指定上层图的相对位置。

**此处使用的是叠图插件**

![](../../images/3.3/3.3-4-1-16.png)

② 如果用户没有上传二维码

则直接添加扣子官方二维码，并按照 ① 中相同的方式叠图

![](../../images/3.3/3.3-4-1-17.png)

官方的二维码图片尺寸是固定的，所以直接按照指定的 30%比例缩放。

**变量聚合**

通过变量聚合节点将两个分支的数据聚合在一起，下游只需要处理同一个变量 output 即可。

![](../../images/3.3/3.3-4-1-18.png)

多个分支的数据只有一个是存在的，直接将其拼接在一起，最终的效果就是执行哪个分支，哪个分支的输出就作为变量聚合后的 output

###### 2.2.2 添加 logo

逻辑和二维码添加相同，略。

![](../../images/3.3/3.3-4-1-19.png)

#### 3. 变量聚合

无论上游是导出链路还是生图链路，最终输出的都是图片，聚合在一起作为 output 传输至下游

![](../../images/3.3/3.3-4-1-20.png)

#### 4. 写入数据库

![](../../images/3.3/3.3-4-1-21.png)

将生成的图片和提示词存储数据表 Generation，该结点会将写入的行号和当前数据表中的数据列表输出至下游。当前链路中输出的数据无用。

##### 数据表

![](../../images/3.3/3.3-4-1-22.png)

官方在当前应用中创建了一个数据表 Generation，字段如上所示。

#### 5. 结束

最终将变量聚合节点的 output 输出，无论执行哪个分支，生成的图片都会被输出。

![](../../images/3.3/3.3-4-1-23.png)

### 4.2 read_history

#### 1. 开始

![](../../images/3.3/3.3-4-2-1.png)

只有一个参数，该参数是用户在本轮对话中输入的内容，然而，在 UI 界面调用该工作流时该参数恒为空。可以理解为开始节点实际上没有参数。

#### 2. 数据库节点

![](../../images/3.3/3.3-4-2-2.png)

从数据表 Generation 查询所有记录的图片、写入时间和提示词，作为列表 outputList 返回，同时返回行号。

#### 3. 结束节点

![](../../images/3.3/3.3-4-2-3.png)

将数据库节点查询到的数据作为 output 返回。
